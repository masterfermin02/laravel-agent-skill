Inertia + Vue — Conventions and example structure

Here's a slice of a typical Inertia + Vue app (single-file components and composables):

resources
├── css
│   └── app.css
└── js
    ├── common
    │   ├── button
    │   │   └── Button.vue
    │   └── card
    │       ├── Card.vue
    │       ├── CardHeader.vue
    │       └── CardContent.vue
    ├── modules
    │   ├── auth
    │   │   ├── useCurrentUser.ts
    │   │   └── Avatar.vue
    │   └── categories
    │       └── CategoryBadge.vue
    ├── pages
    │   ├── layouts
    │   │   └── Layout.vue
    │   ├── profile
    │   │   ├── layouts
    │   │   │   └── ProfileLayout.vue
    │   │   └── ProfilePage.vue
    │   ├── posts
    │   │   ├── components
    │   │   │   └── PublishStatus.vue
    │   │   ├── helpers
    │   │   │   └── generateSlug.ts
    │   │   ├── CreatePostPage.vue
    │   │   ├── EditPostPage.vue

    │   └── DashboardPage.vue
    ├── lib
    │   └── (third-party or generated components)
    └── app.ts

Key conventions (short):

- Directories: kebab-case (e.g. `common`, `posts`, `profile`).
- Component files: PascalCase and one component per `.vue` file (e.g. `Button.vue`).
- Helpers/composables: camelCase `.ts`/.js` files (e.g. `useCurrentUser.ts`).
- Pages: Suffix with `Page` (e.g. `PostsIndexPage.vue`) and ensure default export (using `<script setup>` or `export default`).
- Prefer `<script setup>` + Composition API for new components; use `defineProps`/`defineEmits` for explicit typing.
- Group composables under `common/composables` or `modules/<feature>/composables`.
- Avoid barrel/index files; import components directly using your alias (e.g. `@/common/button/Button.vue`).
- Treat third-party or generated libraries under `resources/js/lib` and wrap them in `resources/js/common` when exposing a project-friendly API.

Minimal Button.vue example (SFC):

<script setup lang="ts">
import { defineProps } from 'vue';

interface Props {
  type?: 'button' | 'submit';
  className?: string;
}

const props = defineProps<Props>();
</script>

<template>
  <button :type="props.type" :class="props.className">
    <slot />
  </button>
</template>

Notes and migration tips:

- If migrating from React: move design primitives to `common`, feature code to `modules`, and pages to `pages` keeping the URL shape.
- Keep autogenerated UI code isolated and wrap it rather than modifying it inline, to make upgrades easier.
